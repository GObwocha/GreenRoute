GreenRoute Technical Architecture

1. System Architecture Overview

The GreenRoute Engine is a high-performance routing application built to optimize vehicle travel for minimum fuel consumption and lowest CO2 emissions. It operates using an In-Memory graph of the Nairobi road network, fetched dynamically via the OSMnx library.

  • Frontend: Django Templates, Vanilla JavaScript, Leaflet Maps, and FontAwesome.
  • Backend: Python, Django framework, and SQLite database.
  • Routing Engine: OSMnx (OpenStreetMap) and NetworkX for graph traversal.

2. Routing Logic (A* Search)

By default, standard routing APIs optimize for absolute shortest distance or fastest time. GreenRoute utilizes a custom eco_weight metric applied to every road segment (edge) in the graph.

Subgraphing Optimization

To ensure queries resolve in milliseconds rather than seconds, the engine creates a dynamic Bounding Box Subgraph around the start and end coordinates. This temporary subgraph reduces the search space significantly.

The A* Algorithm

The backend executes an A* (A-Star) search utilizing a Pythagorean heuristic function. This directs the graph traversal specifically toward the destination coordinate, avoiding exhaustive Dijkstra searches.

3. Eco-Metrics & Personalization

The core objective of GreenRoute is estimating fuel savings. This calculation is derived mathematically from the resulting route distance.

Calculation Formulas

standard_fuel = (distance_km / 100) * fuel_economy_l_per_100km eco_fuel = standard_fuel * (1 - eco_efficiency_gain) fuel_saved = standard_fuel - eco_fuel co2_saved = fuel_saved * 2.31 kg

Administrators can configure the global averages (e.g., Base Fuel Consumption) in the System Settings panel. Furthermore, individual users can override these metrics via the Personalized Vehicle Settings tab on the client dashboard.

4. Live Road Modifiers & Editor

The interactive Map Nodes dashboard allows administrators to actively influence real-time routing decisions by placing geographic penalties.

  • Road Closures: Drops a 50m radius penalty. Any road segments within this radius are marked as restricted.
  • High Traffic Zones: Drops a 200m radius penalty. Any road segments within this radius have their eco_weight heavily multiplied.

These modifiers are saved to the SQLite database and queried instantly during the subgraph generation phase.

5. Extensibility

The platform is designed to scale. The current Nairobi configuration can be swapped for any global city simply by updating the MainConfig.nairobi_graph initialization in apps.py.
